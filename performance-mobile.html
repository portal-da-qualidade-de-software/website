=> Performance

Quando falamos de performance estamos falando de integração entre o hardware e o software, ou seja otimização, a seguir nós listaremos algumas dicas para te ajudar na hora de otimizar seu código.

1) Se você não possui uma medida ou requisito que determine o tempo de resposta esperado, então você também não tem problema de desempenho.

2) Conheça o ambiente em que o sistema executará, tanto a arquitetura física quanto a arquitetura do sistema, pois arquiteturas diferentes demandam implementações e cuidados diferentes.

3) Não concatene Strings, utilize um StringBuilder ou StringBuffer para isso. A exceção para essa regra são as quebras de linha (para melhor entendimento humano) na definição do valor da variável, pois o compilador compilará isso como uma única String.

4) Escolha bem os tipos de dados que serão utilizados, levando em conta que tipos mais precisos ocupam mais memória e tipos primitivos são mais eficientes.

5) Escolha bem a estrutura de dados para a sua aplicação na hora de criar suas entidades e objetos, tenha em mente que você eventualmente buscará esses dados em grandes quantidades e que nem sempre você precisará da entidade inteira para fazer o que você precisa

6) Utilize bem as ferramentas que a linguagem tem a lhe oferecer.

7) Economize recursos e lembre-se que recurso ocioso é recurso desperdiçado.

8) Não crie funções que fazem de tudo, faça funções pequenas que sejam responsáveis por apenas uma atividade.

=> Otimização de Consulta no Banco de Dados

  Para ser eficaz em instruções SQL, o profissional de banco de dados deve ter uma compreensão profunda sobre como o SQL é transformado a partir do código fonte original em uma forma executável.Em um nível alto, o processamento SQL é dividido em várias etapas:

1. Análise do código fonte para localizar erros de sintaxe.

2. Uso do otimizador SQL para obter um plano de execução.

3. O script é executado com base no plano elaborado.

4. Busca do conjunto de resultados do banco de dados e resposta à consulta chamada.

  Geralmente, a maior parte das linguagens de programação executam suas instruções de cima para baixo. Porém, o SQL Server executa em uma ordem única, que é conhecida como fase lógica de processamento de consulta. Estas fases originam uma série de tabelas virtuais com cada uma destas alimentando a fase seguinte (tabelas virtuais ocultas). Estas fases e suas ordens são dadas como observado na Listagem 1.

# Listagem 1. Etapas do processamento lógico do SQL


# A seguir são descritos os itens visualizados na listagem e o que ocorre em cada fase do processamento

1. FROM: um produto cartesiano (cross join) é realizado entre as duas primeiras tabelas na cláusulafrome como resultado a tabela virtual VT1 (tabela virtualizada) é gerada.

2. ON: é aplicado em VT1. Somente as linhas para as quais a é verdadeira serão inseridas em VT2.

2.1. OUTER (junção): se for especificado umouter join(ao contrário de umcross joinou um inner join), as linhas da tabela preservada ou a tabela que não foi encontrada uma correspondência são adicionados às linhas de VT2 como linhas exteriores, gerando o VT3. Se mais de duas tabelas aparecem na cláusulafrom, as etapas 1 a 3 são aplicadas várias vezes entre o resultado da última associação e a próxima tabela na cláusulafrom, até todas as tabelas serem processadas.

3. WHERE: é aplicado a VT3. Apenas as linhas para as quais a é verdadeira são inseridas a VT4.

4. GROUP BY: as linhas da VT4 são organizadas em grupos com base na lista de colunas especificada na cláusula group by, então VT5 é gerado.

5. CUBE | ROLLUP: supergrupos (grupos de grupos) são adicionados às linhas da VT5 gerando VT6;

6. HAVING: é aplicado a VT6. Apenas os grupos para os quais a é verdadeira são inseridos à VT7.

7. SELECT: é processada gerando VT8.

8. DISTINCT: linhas duplicadas são removidas da VT8 gerando a VT9.

9. ORDER BY: as linhas da VT9 são classificadas de acordo com a lista de coluna especificada na cláusulaorder by. Um cursor é gerado (VC10).

10. TOP: o número especificado ou porcentagem de linhas são selecionadas a partir da VC10. A tabela VT11 é gerada e retorna ao executor.

  As etapas básicas descritas para o processamento de uma instrução select se aplicam a outros comandos SQL como insert, update e delete.O processo de identificação dessas linhas é semelhante ao procedimento utilizado para identificar as linhas de origem que colaboram para o conjunto de resultados de uma instrução select.As instruções update e insert podem conter instruções select, incorporadas, que fornecem os valores de dados a serem atualizados ou inseridos.

# Monitoramento de Desempenho

  Monitoramento de desempenho é, de longe, o caminho mais fácil. Para monitorar o desempenho, é suficiente simplesmente identificar os KPIs, métricas e medidas que são importantes para o seu negócio e colocá-los em um painel de controle ou relatório.

Image by storyset on Freepik

  Os sistemas são instrumentados, os dados são coletados e algum ideia é dado para a melhor forma de apresentar os dados para tornar claro o desempenho da organização foi ou é atualmente. Muitos desses sistemas também permitem que os usuários definam "boa" ou "aceitável" de desempenho e, em seguida, enviem alertas quando algo está fora do esperado.

  Mas o que acontece quando algo não é bom, ou não é bom o suficiente, ou se torna o foco de um esforço para melhorar os resultados? Nesse ponto, muitas vezes torna-se problema de outra pessoa para realmente gerenciar o desempenho da organização. O usuário do sistema identifica as pessoas que têm que mudar o seu comportamento, tenta descobrir o que eles estão fazendo, e envolve-os em algum esforço para impulsionar a mudança na organização. Não há nenhuma maneira de alguém olhar para os mostradores e gráficos em seu sistema de gestão de desempenho para fazer alguma coisa sobre eles, o usuário deve deixar o seu painel, sair e dizer a alguém para mudar alguma coisa.

  Para uma organização moderna de qualquer tamanho "a execução da estratégia" significa cada vez mais sistemas de informação. Na performance, mundo 24x7, as operações estão cada vez mais automatizadas. Para alinhar as operações para a execução da estratégia, os sistemas de gestão de desempenho devem estar relacionados com o comportamento de sistemas operacionais. Em particular, eles precisam estar ligados à forma como os sistemas operacionais decidem, e é aí que a gestão decisão entra.

Image by DrawingMyDiary on Freepik

  Para construir esta ligação, nós aplicamos uma abordagem de gerenciamento de decisões simples. Identificamos as decisões, especialmente as decisões operacionais do dia-a-dia, que impactam nossas medidas de desempenho. Isso significa que nós sabemos qual a decisão temos que tomar para mudar se quisermos ter um impacto em uma medida ou KPI. Nós modelamos essas decisões então entendemos como precisamos tomar essas decisões. Identificamos a informação que precisamos e do know-how necessário, e em seguida descrevemos a forma como vamos tomar essas decisões. Nós automatizamos essas decisões, no todo ou em partes, utilizando uma combinação adequada de regras de negócios, análise preditiva e tecnologias de otimização - as tecnologias de sistemas de gerenciamento de decisões.

  Adicionando uma decisão de gestão ao sistema de gestão de desempenho, desta forma garantindo que a estratégia concebida para atingir o desempenho desejado possa ser rapidamente e com precisão aplicado a sistemas de informação. Ao garantir que os resultados das decisões operacionais individuais pode ser medido e melhorado pelos empresários que entendem os objetivos, o gerenciamento de decisão aumenta a agilidade de uma organização e permite que ela se torne adaptável à mudança. Quando análises, especialmente a análise preditiva, são incorporadas nesses sistemas, também podem tornar-se uma organização verdadeiramente analítica.

  Considere um exemplo. Minha dashboard me mostra que a retenção que um segmento da minha base de clientes está caindo. Da minha dashboard eu posso detalhar e descobrir quais os tipos de clientes estão parando, qual o segmento está sendo mais afetado - monitoramento de desempenho. Se minhas decisões são manuais, não modeladas ou ligadas a gestão de desempenho, então estou feito. Tudo que posso fazer é pegar o telefone e falar com alguém.

  Mas se eu estou focado no desempenho de gestão e for identificar, modelar e automatizar as minhas decisões, eu tenho novas opções. Eu posso ver que as decisões têm um impacto sobre a retenção de clientes e revê-los. Posso comparar a forma como eu tomei decisões - as regras de negócio e análises que eu usei - para esse segmento com outros segmentos. Eu posso analisar os clientes no segmento que eu retive e compará-los com os que eu não retive para ver se eu tomaria decisões diferentes. Se eu estou tomando as mesmas decisões, então eu sei que tenho que ajustar a minha tomada de decisão para este segmento porque é claro que não está funcionando bem. Talvez eu precise de uma nova oferta ou de uma avaliação mais agressiva para esse segmento. Porque eu estou usando regras de negócios para apoiar essas decisões, elas são fáceis de modificar e alterar, os mesmos usuários de negócios que veem o problema no sistema de gestão de desempenho podem gerenciar as mudanças de regras si.

Image by vectorjuice on Freepik

  O uso da gestão decisão de automatizar estas decisões-chave age como a cola entre a visibilidade de gerenciamento de desempenho corporativo e de dia para dia os sistemas operacionais. Decisão de gestão é um elemento chave na mudança de monitoramento de desempenho para gerenciamento de desempenho.
